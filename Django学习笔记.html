Django 学习笔记

Django 封装了很多的功能, 主要的目的是为了让开发者专注于功能. 而不是reinvent wheel 该笔记, 主要记录Django学习过程中一些零散的知识.

常用命令

  • Django 使用python manage.py shell来运行shell交互
  • 使用python manage.py shell_plus, 使用ipython进行交互. 需要安装django_extensions
  • 使用python manage.py test <appname>来执行单元测试
  • 使用django-admin startproject mysite来创建一个django应用
  • 使用django-admin startapp app来创建一个django app
  • 使用python manage.py runserver来运行server
  • 使用python manage.py createsuperuser来创建超级管理员
  • 使用python manage.py makemigrations <app_label>用来记录当前关于数据库的所有改动
  • 使用python manage.py migrate即将改动执行在对应的数据库上面
  • 使用python manage.py sqlmigrate <app_name> <migration_name>来查看具体某个migration的SQL信息

常见问题

  • 使用virtualenv -p python3 <envname> 用来创建python3的虚拟环境

  • 创建外键Foreign key, 即在创建了两个不同表之间的关联. Django中ORM通过choice_set来获取与之关联的对象. choice_set即为RelatedManager, 可以通过choice_set来进行一系列的query

  • 对于model层的操作, 如果涉及返回HTTP异常码, 则可以使用类似Django封装好的get_object_or_404方法.

  • Django提供了csrf token CSRF(Cross Site Request Forgery) 跨站请求伪造, 该请求可以使用构造生成csrf token来避免. 即在URL上面添加csrf的信息.

  • DTL为Django template language, Django的模板系统主要是用来渲染html, 用来展示的不应该用来处理程序逻辑.

  • Template inheritance模板继承, Django的模板继承允许定义基础骨架模板, 包含所有的共有的元素, 然后再定义不同的block使用.

  • Django reverse和HttpResponseRedirect搭配使用, 用来重定向请求页面.

  • Django项目中可以包含很多个app, Django通过manage.py来管理整个app, 因此最外层的带有manage.py的部分不能添加__init__.py

  • Django 提供了一个测试客户端用来模仿真实用户在view层面的交互. 通常使用from django.test import Client来进行构造客户端

  • Django reverse为Django模板中的逆向解析, 即通过视图的名字, 和参数逆向获得请求的URL

  • Django model filter使用双下划线, 双下划线主要用来连接具体的动作

  • Django admin.site.register用来注册model, 和model admin_class

  • Django ListView的继承关系如下: 主要是由三个基础类继承而来, 分工明确

    • View基础功能类, 主要负责处理HTTP, GET或者POST的请求
    • ContextMixin基础混合类, 主要负责从模板中读取所需要的参数
    • TemplateResponseMixin基础混合类, 主要负责渲染模板的功能
    • MultipleObjectMixin主要用来从数据库获取多条信息

  • Django通过urlconf来处理路由关系:

    • Django在最初的原始地方定义了urlconf文件
    • 在其他app中可以定义相对应的urlconf, 需要在root urlconf中添加include进行包含
    • Django在加载python模块时, 查找模块中的urlpatterns变量进行映射对于的url, urlpattern是path或者re_path的list

  • (\P<name>[\w-]+) 该正则表达式在Django中传递name关键字参数. 是正则表达式的named capture group

  • Django slug: 即构造合法URL的一种方法, 例如: Hello world => Hello-world, 当slug作为URL是, 可以使用自增主键和slug组合一起的方式, 保证了URL的可读性, 通过ID数字来存取又保证了灵活性.

    • URL:url(r'^(?P<slug>[-\w\d]+),(?P<id>\d+)/$', view=myviews.article, name='article')
    • Path:path('<slug:slug>,<int:id>/', views.article_detail, name='article')

  • Django中url为带正则匹配的url, 通常使用^xxxx$对url的开头和结尾进行匹配

  • Django通过login_required来控制登录状态

  • Django通过user_passed_test当返回为False时,来控制重定向

  • Django使用permission_required来控制用户是否有某一个具体的权限, 主要是在于与model层的控制

  • Django中Model的Meta类为一个内部类, 用来定义Model的行为, 比如定义db_table 数据库的表明, app_label 具体某一个app, 仅当当前model不在对应的app下面时使用

  • Django permission可以通过ModelAdmin提供对应的方法比如has_add_permission来实现. 自定义的permission可以直接在Model的Meta中实现, 还可以直接创建permission, 先获得ContentType, 再为该ContentType创建permission

  • Django unittest 对 model 进行测试时, 不会对于生产环境的数据库进行修改, 而是创建一个专门为测试环境的数据库, 不管测试是否通过, Django都会删除数据库.

  • Django cache配置redis作为backend之后, 通过Django的cache, 放入redis, 会经过make_keys的函数, 即返回带version的keys. 例如, 通过Django cache进行cache操作key:hello=> :1:hello

  • Django支持4种session引擎:

    • 数据库的session
    • 缓存的session: 有更加优秀的性能, 如果是仅使用cached-based, 而且是memcached, 这样由于memcache不会很长时间保存数据, 这种情况下选择db cache好. 而且本地local memcached是线程不安全的. 使用redis cache应该没有问题.
    • 文件基础的session
    • cookie的session

  • Django中signed cookie即签名cookie, 服务端通过共同的算法相信客户端的cookie, 而且服务不存放任何的session信息

  • Django中登录认证, 可以使用signed cookie也可使用session的一套

    • signed cookie: 主要使用两个方法. request.get_signed_cookie和response.set_signed_cookie, 使用cookie的弊端是容易受到xss.
    • session 主要使用session_id. session中可以放置其他的一些方法

  • HTTP 401: Unauthorized: 主要是对于Authentication error, 通常为登录验证失败

  • HTTP 403: Forbidden: 主要用于No such permission, 通常为没有对应的权限

  • 服务端通常为先Authenticate=>Authorize

  • Django中通过继承并重写BaseAuthentication的authenticate方法来获取自定义的User进行认证

  • Django中通过继承并重写BasePermission的has_permission方法来对特定的请求进行鉴权

  • Django Rest Framework中通过继承并重写BaseAuthentication的authenticate方法来获取自定义的User进行认证

  • Django Rest Framework中通过继承并重写BasePermission的has_permission方法来对特定的请求进行鉴权

  • Django Rest Framework即DRF: 类似于Flask更加适合开发api

  • Django更加适合开发web app

  • Django中的request.user由函数get_user_model()来获得. get_user_model通过settings.AUTH_USER_MODEL来获得自定义的CustomUser

  • Django中的authenticate(request=None, **credentials)函数, 根据AuthenticationBackend来获取UserModel,通过UserModel来对credentials进行验证. 如果某一个AuthenticationBackend抛出PermissionDenied则退出后续校验, 如果返回None则继续进行其他的校验. 如果所有的Authentication失败, 则发出用户认证失败的信号user_login_failed.send(...)

  • Django使用login和logout两个函数来进行user的登录和登出, 使用Django Session framework来保存user的ID. login: 将已登录的user ID保存到Django Session中, logout: 清除某个user的session信息, 当该用户不存在时, 也不会抛出异常.

  • Djnaog login如果多次login则会在缓存中产生多个session信息.

  • Django signed_cookie需要设置HttpOnly来防止客户脚本来获取cookie, 即防止XSS, Cross-site Scripting

  • Django cookie session vs Django datebase session

    • cookie session 存贮用户的k-v在客户端, 当服务器是一个集群时, 可以平滑的操作, 即用同一个Hasher+salt算法, 就可以获得cookie session的信息.
    • cookie session 存贮在客户端的cookie有大小限制, 最大不超过4kb,存贮信息有限.
    • 存储在客户端浏览器的cookie可以被服务端设置expires很长, 甚至几个月
    • cookie session 使用时必须使用HttpOnly Flag, 和设置相应的安全标记, 以防cookie被XSS 盗用
    • database session存贮用户的信息在服务端, 当服务端没有设置中心存储时, 如果一个用户第一次登录了集群中的某一台机器时, 该机器就会保存相应的session信息.
    • database session客户端不需要发送额外的信息, 只需要在每次请求时发送一个特定的session id
    • database session可以存储用户的敏感信息.

  • cookie max_age表示cookie的存活时间

  • cookie expires表示cookie的最终期限

  • 如果同时设置cookie 的 max_age 和 expire 大部分浏览器会选择支持max_age, IE浏览器会选择支持expire

  • Django LoginRequiredMixin设计思想

    • 首先继承自基类AccessMixin
    • 重写BaseViewView基类中的, dispatch方法, 即先处理认证相关的信息, 然后调用super().dispatch方法进行处理请求.

  • CSRF Cross-site request forgery跨站请求伪造, 两个必要条件

    • 登录受信任网站A, 在本地生产cookie
    • 在不登出信任网站A的情况下, 访问危险网站B. B有可能要求访问A, 则该访问携带了cookie.

  • 避免和防范CSRF, 即使用CSRF token, 客户端从服务端获取CSRF token, 然后客户端提交表单时, 同时提交CSRF tokekn. 服务端对CSRF token进行校验.CSRF token很大程度上面降低了CSRF情况, 但是XSS的情况, 黑客直接劫持了cookie则可以再次发起CSRF攻击.

  • gunicorn是WSGI server, gevent是python coroutine

  • HMAC (keyed-hash message authentication code)是一种特定的MAC(message authentication code), 即对secret key和message进行hash, 通常作为获取认证信息的hash值. 在Django中用来构造sessionID, 其hash function为SHA-1

  • django-rest-framework serialize mulit/part file时直接使用serialzier.data会导致file变成None, 解决方法是使用validated_data

  • django 中cookie 的值为32位小写字母与数字的组合，cookie hash=> 0-100 简单的实现是将cookie字符串按照32进制转换即可，然后可以根据数字进行一些额外的操作

  • Middleware的概念在django中有很多地方使用，且设计优良，例如django的cookie-session机制，使用了Middleware对请求header中的cookie参数，并初始化load特定的session存储 request.session = SessionStore(session_key)，这样的话在业务层直接使用request.session即可。且Middle封装了对返回Response时的一些处理，很合理。

  • 自定义Middleware 使用时需要重写process_request和process_response, 实现自定义的一些需求
