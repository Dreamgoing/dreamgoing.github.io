简介

背景

在过去, 通常编写原生的SQL语句, 这样会引发如下的问题, 原生的SQL语句难以维护, 而且是易于出错的.

关系对象映射(ORM)

Object-Relational Mapping 是一种编程技巧, 用来在各种不兼容平台中做数据类型的转换, 它能够自动的将存储在关系性数据库表中的数据转化成在应用程序中更加常用的对象.

使用原生sql

import sqlite3
conn = sqlite3.connect('example.db')
 
c = conn.cursor()
c.execute('''
          CREATE TABLE person
          (id INTEGER PRIMARY KEY ASC, name varchar(250) NOT NULL)
          ''')
c.execute('''
          CREATE TABLE address
          (id INTEGER PRIMARY KEY ASC, street_name varchar(250), street_number varchar(250),
           post_code varchar(250) NOT NULL, person_id INTEGER NOT NULL,
           FOREIGN KEY(person_id) REFERENCES person(id))
          ''')
 
c.execute('''
          INSERT INTO person VALUES(1, 'pythoncentral')
          ''')
c.execute('''
          INSERT INTO address VALUES(1, 'python road', '1', '00000', 1)
          ''')
 
conn.commit()
conn.close()

使用python SQLAIchemy

SQLAIchemy有如下几个重要的点:

  • 一个Table代表数据中一个实际的表
  • 一个mapper映射一个python的class到实际的数据表中
  • 一个class object定义了一个数据库中记录的信息映射到一个普通的python object中

如下是使用python SQLAIchemy的例子

import os
import sys
from sqlalchemy import Column, ForeignKey, Integer, String
from sqlalchemy.ext.declarative import declarative_base
from sqlalchemy.orm import relationship
from sqlalchemy import create_engine
 
Base = declarative_base()
 
class Person(Base):
    __tablename__ = 'person'
    # Here we define columns for the table person
    # Notice that each column is also a normal Python instance attribute.
    id = Column(Integer, primary_key=True)
    name = Column(String(250), nullable=False)
 
class Address(Base):
    __tablename__ = 'address'
    # Here we define columns for the table address.
    # Notice that each column is also a normal Python instance attribute.
    id = Column(Integer, primary_key=True)
    street_name = Column(String(250))
    street_number = Column(String(250))
    post_code = Column(String(250), nullable=False)
    person_id = Column(Integer, ForeignKey('person.id'))
    person = relationship(Person)
 
# Create an engine that stores data in the local directory's
# sqlalchemy_example.db file.
engine = create_engine('sqlite:///sqlalchemy_example.db')
 
# Create all tables in the engine. This is equivalent to "Create Table"
# statements in raw SQL.
Base.metadata.create_all(engine)

代码分析:

  • declarative_base(): 构造一个基类, 用来生成陈述性定义的类. 举例即为 Table
  • create_engine(): 创建一个连接具体数据库的存储引擎
  • Base.create_all(): 为创建所有的表
  • sessionmaker(bind=engine) 来创建一个工厂类, ORM与数据库的查询和交互都需要通过session来进行
  • db.session.commit 和 db.session.rollback 来提交和回滚事务
  • session 可以理解为一次事务, commit则为提交一次事务, rollback则为回滚一次事务

具体的细节部分可以参考官方文档

SQLAIchemy性能分析

问题: 使用ORM插入比较大量的数据时, ORM会变得非常非常的慢

如下是不同的插入方式进行插入数据时的性能比较结果

SQLAlchemy ORM: Total time for 10000 records 3.96545100212 secs
SQLAlchemy ORM pk given: Total time for 10000 records 1.55196404457 secs
SQLAlchemy ORM bulk_save_objects(): Total time for 10000 records 0.000293970108032 secs
SQLAlchemy ORM bulk_insert_mappings(): Total time for 10000 records 0.00104308128357 secs
SQLAlchemy Core: Total time for 10000 records 0.2213139534 secs
sqlite3: Total time for 10000 records 0.0422978401184 sec

原因: SQLAIchemy ORM同步改变到数据库时, 使用单元化的工作模式(unit of work). 这种工作模式, 不仅仅是简单的插入数据. 这包括了一系列的操作: 主要是ORM系统对于对象的跟踪, 和周期性地将待改变的内容传送的数据库中.

unit of work: ORM跟踪维护事务影响的对象列表, 主要是为了解决读写一致性和并发的问题

Maintains a list of objects affected by a business transaction and coordinates the writing out of changes and the resolution of concurrency problems.

解决方案:

  • 使用bulk_insert, 是一种非常高校的解决方案
  • 使用原生的SQL语句, 经大致测试, 比SQLAIchemy的unit of work

SQLAIchemy架构

架构图解

SQLAIchemy 将主要的任务拆分成两个:

  1. Core : 包括Python Database 交互, 对于文本模式SQL的表达和处理, 使得数据库能够理解和处理, 数据库模式的管理
  2. ORM : 在Core的上层, 是在Core的基础上对于数据库的抽象

设计优点:

  1. Core/ORM 在SQLAIchemy中的分离是非常好的设计, 显式的Core设计使得ORM关注于database-mapped class, 而不是直接将string column传递给数据库
  2. 当提供SELECT查询时, 使用select数据结构而不是直接根据database-mapped class中的attributes 来构造SQL字符串
  3. 当获取结果时, 通过ResultProxy 来清晰地映射结果行, 而不是直接从数据库中转换直接转换结果到预定义的对象中
